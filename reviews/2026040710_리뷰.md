## 2. 잘한 점

### 재사용 가능한 헤더 컴포넌트 설계

- `header.js`에서 `renderHeader(targetSelector)` 함수로 헤더를 재사용 가능하게 만든 건 **정말 잘했어요!** 바닐라 JS에서 컴포넌트처럼 동작하는 코드를 만든 거예요.
- `main_list.js`에서 `renderHeader("#header-container")`로 호출하는 방식이 깔끔해요.

```js
// header.js — 어디서든 호출 가능한 재사용 컴포넌트!
export function renderHeader(targetSelector = "body") { ... }

// main_list.js — 다른 페이지에서 한 줄로 헤더 추가
renderHeader("#header-container");
```

### API 코드와 화면 코드 분리

- `src/API/main_list.js`에 API 호출 로직을 분리하고, `src/main/main_list/main_list.js`에서 화면 렌더링만 담당하는 구조가 **매우 좋아요!** 이것이 바로 **"관심사의 분리"**라는 중요한 원칙이에요.
- API 함수가 `async/await`와 `try/catch`를 사용해서 에러 처리도 잘 되어있어요.

```js
// API 호출 (src/API/main_list.js) — 데이터만 담당
export const getMainList = async () => { ... };

// 화면 렌더링 (src/main/main_list/main_list.js) — 화면만 담당
function renderSmallCard(post) { ... }
```

### 디자인 토큰(CSS 변수) 시스템 도입

- `theme.css`에 색상값을 CSS 변수로 정의한 건 **프로 수준의 접근**이에요! 나중에 색상을 바꿀 때 한 곳만 수정하면 돼요.

```css
/* theme.css — 색상을 한 곳에서 관리 */
:root {
  --background-base: #171717;
  --color-primary: #dc2626;
  --color-error: #dc2626;
  --color-success: #16a34a;
}
```

### Input 컴포넌트의 접근성(A11y) 처리

- `input.html`에서 `aria-labelledby`, `aria-describedby`, `aria-label`, `aria-pressed` 같은 접근성 속성을 꼼꼼하게 넣은 건 **정말 인상적이에요!** 시각 장애가 있는 사용자도 이 폼을 사용할 수 있어요.
- `<label for="...">` 연결도 정확하게 되어있고, SVG 아이콘에 `aria-hidden="true"`를 붙인 것도 완벽해요.

### Input 유효성 검사 로직의 깔끔한 구조

- `input.js`에서 규칙(rules) 객체로 유효성 검사를 관리하는 패턴이 아주 잘 설계되었어요. 새로운 필드를 추가할 때 rules 객체에 한 줄만 추가하면 되거든요!

```js
const rules = {
  username: { regex: /^[a-z0-9_]{4,20}$/, hint: "...", error: "...", ok: "..." },
  nickname: { regex: /^[^\n]{1,10}$/,      hint: "...", error: "...", ok: "..." },
  password: { regex: /^(?=.*[A-Z])...$/,    hint: "...", error: "...", ok: "..." },
};
```

### 반응형 디자인 적용

- `header.css`와 `main_list.css` 모두 `@media (min-width: 768px)` 미디어 쿼리를 사용해서 모바일/데스크탑 대응을 잘 하고 있어요.
- 특히 헤더의 검색바가 모바일에서는 전체 너비, 데스크탑에서는 중간에 배치되는 구현이 깔끔해요.

### 컴포넌트 프리뷰 페이지

- `button.html`과 `input.html`에 "구조 복붙용 컴포넌트 파일입니다"라는 주석과 함께 프리뷰 페이지를 만든 건 **팀 협업에 아주 유용한 방법**이에요! 디자인 시스템을 만드는 첫걸음이에요.

### BEM 네이밍 규칙 사용

- CSS 클래스명에 `featured-card__poster`, `movie-card__info`, `genre-section__header` 같은 **BEM(Block-Element-Modifier)** 패턴을 사용한 건 훌륭해요! CSS가 훨씬 예측 가능하고 충돌이 적어져요.

## 3. 개선하면 좋을 점

### 3-1. API 응답 데이터를 innerHTML에 직접 넣으면 보안에 취약해요 (중요도: 높음)

**현재 상태:**

```js
// src/main/main_list/main_list.js 39번째 줄
function renderFeaturedCard(post) {
  return `
    <h2 class="featured-card__title">${post.title}</h2>
    <img src="${post.imageUrl}" alt="${post.title}" />
  `;
}

// 104번째 줄 — innerHTML로 삽입
latestSectionEl.innerHTML = renderLatestSection(latest);
```

**왜 개선하면 좋을까?**

서버에서 받아온 데이터(`post.title`, `post.imageUrl` 등)를 **그대로 HTML에 넣으면**, 만약 악의적인 사용자가 제목에 `<script>alert('해킹!')</script>` 같은 코드를 넣었을 때 그게 실제로 실행될 수 있어요!

이것을 **XSS(Cross-Site Scripting) 공격**이라고 해요. 웹 보안에서 가장 흔하고 위험한 공격 중 하나예요.

> **쉽게 말하면:** 편지를 받아서 그 내용을 그대로 게시판에 붙이면, 누군가 거기에 "이 게시판을 지워라"라는 명령을 적어 보낼 수 있어요. 내용을 먼저 검수(살균)한 후에 붙여야 안전해요.

**추천:**

```js
// 텍스트를 안전하게 변환하는 함수를 만들어주세요
function escapeHTML(str) {
  const div = document.createElement("div");
  div.textContent = str;
  return div.innerHTML;
}

// 사용 예시
function renderFeaturedCard(post) {
  const title = escapeHTML(post.title);
  const genre = escapeHTML(GENRE_MAP[post.genre] ?? post.genre ?? "");

  return `
    <a href="/src/main/detail/detail.html?postId=${post.postId}" class="featured-card">
      <div class="featured-card__poster">
        ${post.imageUrl ? `<img src="${escapeHTML(post.imageUrl)}" alt="${title}" />` : ""}
      </div>
      <div class="featured-card__info">
        <h2 class="featured-card__title">${title}</h2>
      </div>
    </a>
  `;
}
```

### 3-2. accessToken이 콘솔에 출력되고 있어요 (중요도: 높음)

**현재 상태:**

```js
// src/API/main_list.js 4번째 줄
console.log(`accessToken: ${localStorage.getItem("accessToken")}`);
```

**왜 개선하면 좋을까?**

**Access Token은 로그인한 사용자의 "열쇠"**에요. 이걸 콘솔에 출력하면:
- 다른 사람이 개발자 도구(F12)에서 토큰을 볼 수 있어요
- 누군가 이 토큰을 복사하면 **당신인 척 하고** API를 호출할 수 있어요

개발 중 디버깅용으로 넣은 거라면 이해가 되지만, 반드시 배포 전에 제거해야 해요!

**추천:**

```js
// 수정 전 (src/API/main_list.js 3~4번째 줄)
console.log("Fetching main list from API...");
console.log(`accessToken: ${localStorage.getItem("accessToken")}`);

// 수정 후 — 디버그 로그 모두 제거
// (디버그가 필요하면 브라우저 개발자 도구의 Network 탭을 사용하세요!)
```

> **쉽게 말하면:** 집 열쇠를 복사해서 아무나 볼 수 있는 게시판에 붙여놓는 것과 같아요. 열쇠는 남에게 보여주면 안 돼요!

### 3-3. `.env` 파일이 `.gitignore`에 없어요 — 긴급! (중요도: 높음)

**현재 상태:**

```
# .gitignore (현재)
node_modules/
dist/
build/
```

```
# .env (현재 — Git에 올라갈 수 있음!)
VITE_API_BASE_URL=https://api.fullstackfamily.com/api/buildup/v1
```

**왜 개선하면 좋을까?**

`.env` 파일은 **환경 변수**를 저장하는 파일이에요. 지금은 API URL만 들어있지만, 나중에 비밀 키(API Key)나 데이터베이스 비밀번호 같은 민감한 정보가 들어갈 수 있어요.

`.gitignore`에 없으면 Git에 올라가서 **GitHub에 공개**될 수 있어요! 이전 리뷰(3-5)에서 이미 언급했는데, 지금 `.env` 파일이 실제로 존재하니까 **더더욱 긴급**해요.

**추천:**

```gitignore
# .gitignore — 수정 후

# 환경 변수 (민감 정보 보호!)

.env.local
.env.production

# OS 파일

# 에디터 설정
.vscode/

```

그리고 팀원들이 `.env` 파일의 구조를 알 수 있도록 `.env.example` 파일을 만들어주세요:

```bash
# .env.example (Git에 올려도 되는 예시 파일)
VITE_API_BASE_URL=https://여기에-API-주소를-넣으세요
```

> **쉽게 말하면:** `.env`는 비밀 일기장이에요. `.gitignore`에 안 넣으면 학교 게시판(GitHub)에 비밀 일기가 올라가는 거예요!

### 3-4. header.js에서 대상 요소가 없을 때 에러가 나요 (중요도: 중간)

**현재 상태:**

```js
// src/main/header/header.js 31~32번째 줄
const target = document.querySelector(targetSelector);
target.insertAdjacentHTML("afterbegin", html);
```

**왜 개선하면 좋을까?**

`document.querySelector()`는 해당 요소를 찾지 못하면 `null`을 반환해요. 만약 오타로 `renderHeader("#headr-container")` 같이 호출하면, `null.insertAdjacentHTML(...)`에서 **에러가 터져요.**

이런 에러는 화면이 아예 안 나오게 만들 수 있어서, **방어적인 코드**를 넣어주는 게 좋아요.

**추천:**

```js
// 수정 전
const target = document.querySelector(targetSelector);
target.insertAdjacentHTML("afterbegin", html);

// 수정 후
const target = document.querySelector(targetSelector);
if (!target) {
  console.error(`헤더를 넣을 요소를 찾을 수 없어요: "${targetSelector}"`);
  return;
}
target.insertAdjacentHTML("afterbegin", html);
```

> **쉽게 말하면:** "3번 교실에 공지를 붙여줘"라고 했는데 3번 교실이 없으면? 그냥 멈추지 말고 "3번 교실이 없어요!"라고 알려주는 게 좋아요.

### 3-5. Vite에서 `public/` 폴더 안 파일 경로가 잘못되어 있어요 (중요도: 중간)

**현재 상태:**

```js
// src/main/header/header.js 8번째 줄
<img src="/public/Bui1dBox.png" alt="빌드업 텍스트 로고" />
<img src="/public/Search_icon.svg" alt="검색" />
```

**왜 개선하면 좋을까?**

Vite에서 `public/` 폴더에 넣은 파일은 **자동으로 루트(`/`)에서 접근**할 수 있어요. 즉:
- `public/Bui1dBox.png` → 경로는 `/Bui1dBox.png`
- `public/Search_icon.svg` → 경로는 `/Search_icon.svg`

`/public/Bui1dBox.png`이라고 쓰면 실제로는 `public/public/Bui1dBox.png`을 찾게 돼서 **이미지가 안 보여요!**

// 수정 전

### 3-6. header.html과 header.js의 HTML이 서로 달라요 — 응집도 문제 (중요도: 중간)

**현재 상태:**

```html
<!-- header.html 19번째 줄 — 로고 링크 경로 -->
<a href="/main_list.html" class="header-logo-link">

<!-- header.js 6번째 줄 — 로고 링크 경로 (다름!) -->
<a href="/src/main/main_list/main_list.html" class="header-logo-link">
```

**왜 개선하면 좋을까?**

같은 헤더인데 **두 군데에서 HTML을 따로 관리**하고 있어요. 이러면:
1. 헤더를 수정할 때 **두 파일을 모두 수정**해야 해요 (잊어버리기 쉬워요!)
2. 이미 경로가 서로 달라져 있어요 — 실수가 발생한 거예요
3. **응집도(Cohesion)**가 낮아져요. 하나의 기능은 한 곳에서 관리해야 해요.

**응집도**란 "관련 있는 코드가 한 곳에 모여있는 정도"예요. 같은 헤더 HTML이 두 파일에 흩어져 있으면 응집도가 낮은 거예요.

**추천:**

`header.html`은 **프리뷰 전용**임을 명확히 하고, 실제 헤더는 `header.js`만 사용하세요:

```html
<!-- header.html 상단에 주석 추가 -->
<!-- ⚠️ 이 파일은 프리뷰/테스트 전용입니다.
     실제 헤더는 header.js의 renderHeader()를 사용하세요.
     이 파일을 수정하면 header.js도 함께 수정해야 합니다! -->
```

더 좋은 방법은 `header.html`에서도 `header.js`를 import해서 동일한 코드를 사용하는 거예요:

```html
<!-- header.html — header.js를 사용하는 프리뷰 -->
<body>
  <div id="header-container"></div>
  <script type="module">
    import { renderHeader } from './header.js';
    renderHeader('#header-container');
  </script>
</body>
```

### 3-7. package.json에 같은 패키지가 dependencies와 devDependencies 양쪽에 있어요 (중요도: 중간)

**현재 상태:**

```json
{
  "devDependencies": {
    "@tailwindcss/vite": "^4.2.2",
    "tailwindcss": "^4.2.2",
    ...
  },
  "dependencies": {
    "@tailwindcss/vite": "^4.2.2",
    "tailwindcss": "^4.2.2"
  }
}
```

**왜 개선하면 좋을까?**

`@tailwindcss/vite`와 `tailwindcss`가 **두 곳에 중복**으로 들어있어요.

- `dependencies` = 실제 서비스에 필요한 패키지 (예: 서버에서 돌아가는 라이브러리)
- `devDependencies` = 개발할 때만 필요한 패키지 (예: 빌드 도구, 테스트 도구)

Tailwind CSS와 Vite는 **빌드 도구**예요. 빌드하면 CSS 파일이 만들어지니까, 서비스할 때는 Tailwind 자체가 필요 없어요. 그래서 `devDependencies`에만 있으면 돼요.

**추천:**

```json
{
  "devDependencies": {
    "@tailwindcss/vite": "^4.2.2",
    "eslint": "^10.1.0",
    "jest": "^30.3.0",
    "tailwindcss": "^4.2.2",
    "vite": "^8.0.3"
  }
}
```

`dependencies` 블록은 통째로 삭제하세요. 터미널에서 이렇게 하면 돼요:

```bash
npm uninstall @tailwindcss/vite tailwindcss
npm install -D @tailwindcss/vite tailwindcss
```

> **쉽게 말하면:** 요리(빌드)할 때 쓰는 도구(프라이팬)와 손님에게 내는 음식(결과물)은 달라요. Tailwind는 "프라이팬"이지 "음식"이 아니에요. 주방(devDependencies)에만 두면 돼요!

### 3-8. CSS 색상값을 직접 쓰지 말고 theme.css의 CSS 변수를 사용하세요 — 결합도 문제 (중요도: 중간)

**현재 상태:**

```css
/* theme.css에서 이미 정의된 변수들 */
:root {
  --background-base: #171717;
  --background-surface: #262626;
  --text-primary: #ffffff;
  --text-secondary: #737373;
}

/* 그런데 main_list.css에서는 직접 색상값을 씀 */
body { background-color: #171717; }           /* → var(--background-base) */
.featured-card__poster { background-color: #404040; }
.featured-card__title { color: #ffffff; }     /* → var(--text-primary) */
.featured-card__meta { color: #d4d4d4; }      /* → var(--text-tertiary) */
.featured-card__crew { color: #a3a3a3; }
.genre-section__title { color: #ffffff; }     /* → var(--text-primary) */
.movie-card { background-color: #262626; }    /* → var(--background-surface) */
```

**왜 개선하면 좋을까?**

`theme.css`에 CSS 변수를 잘 정의해놨는데, 정작 다른 CSS 파일에서 **직접 색상값(`#171717`)을 쓰고 있어요**. 이러면:
1. 나중에 다크모드/라이트모드 전환 같은 테마 변경이 **불가능**해요
2. 색상을 바꾸려면 **모든 파일을 찾아서 일일이 수정**해야 해요
3. **결합도(Coupling)**가 높아져요 — 색상 변경이 여러 파일에 영향을 줘요

**결합도**란 "한 곳을 수정했을 때 다른 곳도 수정해야 하는 정도"예요. 색상값을 직접 쓰면 결합도가 높아지고, CSS 변수를 쓰면 theme.css 한 곳만 바꾸면 되니까 결합도가 낮아져요.

**추천:**

```css
/* 수정 전 */
body { background-color: #171717; color: #ffffff; }

/* 수정 후 — CSS 변수 사용 */
body { background-color: var(--background-base); color: var(--text-primary); }
```

`header.css`도 마찬가지예요:

```css
/* 수정 전 */
.site-header { background: var(--neutral-900, #171717); }

/* 수정 후 — theme.css의 변수와 일치시키기 */
.site-header { background: var(--background-base, #171717); }
```

> **쉽게 말하면:** 색상을 직접 쓰는 건 모든 방에 페인트 색을 따로 적어둔 것과 같아요. CSS 변수는 "메인 색상표"를 만들어놓고 "1번 색으로 칠해"라고 하는 거예요. 색을 바꾸고 싶으면 색상표 하나만 수정하면 돼요!

### 3-9. main_list.css에서 Tailwind를 import하지만 거의 사용하지 않아요 (중요도: 낮음)

**현재 상태:**

```css
/* src/main/main_list/main_list.css 1번째 줄 */
@import "tailwindcss";
```

그런데 이 파일 전체가 **순수 CSS로만 작성**되어 있고, Tailwind 클래스를 사용하는 곳이 없어요.

**왜 개선하면 좋을까?**

`@import "tailwindcss"`는 **Tailwind의 모든 기본 스타일(reset 등)**을 가져와요. 필요 없는 코드를 불러오면 CSS 파일 크기가 커지고, 가끔 **Tailwind의 기본 리셋 스타일이 직접 작성한 CSS와 충돌**할 수 있어요.

또한 프로젝트에서 Tailwind 사용 방식이 파일마다 달라요:
- `main_list.css` → `@import "tailwindcss"` (전체 import)
- `header.css` → `@import "tailwindcss"` (전체 import)
- `button.css` → `@reference "tailwindcss"` (참조만)
- `input.css` → `@reference "tailwindcss"` (참조만)

**추천:**

**방법 A:** Tailwind 클래스를 사용하지 않는 파일에서는 `@import "tailwindcss"`를 제거하세요.

**방법 B:** 팀에서 Tailwind 사용 규칙을 정하세요.
- "Tailwind 클래스를 HTML에서 쓸 거면 해당 CSS에 `@import` 추가"
- "CSS 파일에서 `@apply` 같은 Tailwind 기능만 쓸 거면 `@reference` 사용"

```css
/* Tailwind 클래스를 사용하지 않으면 → 제거 */
/* @import "tailwindcss"; ← 삭제 */

/* Tailwind @apply 등을 쓰려면 → @reference */
@reference "tailwindcss";
```

> **쉽게 말하면:** 영어 사전을 가방에 넣고 다니는데 영어 수업이 없는 날이에요. 무거운 사전은 빼고, 필요한 날만 가져가면 가방이 가벼워져요!

### 3-10. input.js가 `type="module"` 없이 로드돼요 (중요도: 낮음)

**현재 상태:**

```html
<!-- src/components/input.html 133번째 줄 -->
<script src="./input.js"></script>

<!-- 반면 main_list.html 28번째 줄 -->
<script type="module" src="./main_list.js"></script>
```

**왜 개선하면 좋을까?**

`type="module"`이 없으면:
- `import`/`export` 문법을 사용할 수 없어요
- 변수가 **전역(global)**으로 노출돼요 (다른 스크립트와 충돌 가능)
- `strict mode`가 자동 적용되지 않아요

지금은 `input.js`에서 `import`를 안 쓰니까 동작은 하지만, 나중에 `input.js`를 다른 페이지에서 재사용하려면 `type="module"`이 필요해요.

**추천:**

```html
<!-- 수정 전 -->
<script src="./input.js"></script>

<!-- 수정 후 -->
<script type="module" src="./input.js"></script>
```

> **쉽게 말하면:** `type="module"`은 "이 코드는 독립적인 방에서 실행해줘"라는 뜻이에요. 없으면 모든 코드가 한 방에서 섞여서 이름이 충돌할 수 있어요.
