## 2. 잘한 점

### 프로젝트 구조 설계

- **페이지별 폴더 분리가 아주 잘 되어 있어요!** `account/login/`, `main/detail/` 등 기능별로 폴더를 나누고, 각 페이지의 HTML과 CSS를 같은 폴더에 두는 방식은 실무에서도 많이 쓰는 좋은 패턴이에요.
- **API 호출 코드를 `src/API/` 폴더에 분리한 것도 훌륭해요.** 화면 코드(HTML/CSS)와 데이터를 가져오는 코드(API)를 분리하면 나중에 유지보수가 훨씬 쉬워져요. 처음 프로젝트를 만드는데 이런 구조를 잡은 건 정말 대단합니다!

### Vite 빌드 설정

- `vite.config.js`에서 **멀티 페이지(MPA) 설정**을 잘 해놨어요. 모든 HTML 페이지를 `rollupOptions.input`에 등록한 건 Vite를 처음 써보는 분이 하기엔 꽤 어려운 작업인데, 잘 해냈어요!
- `@` alias 설정도 해뒀는데, 이건 `import`할 때 긴 경로 대신 `@/API/login.js`처럼 짧게 쓸 수 있게 해주는 편리한 기능이에요. 좋은 설정이에요!

```js
// vite.config.js에서 잘 설정한 부분
resolve: {
  alias: {
    "@": path.resolve(__dirname, "./src"),
  },
},
```

### 루트 index.html 기본기

- `<!doctype html>`, `lang="ko-KR"`, `charset="UTF-8"`, `viewport` 메타태그를 모두 넣어준 건 HTML 기본기를 잘 알고 있다는 증거예요!
- 한국어 사이트에 맞게 `lang="ko-KR"`을 정확히 써준 점도 좋아요.

### Git 협업 환경 준비

- `.github/PULL_REQUEST_TEMPLATE.md`와 Issue Template을 만든 건 팀 협업을 위한 훌륭한 준비예요.
- 커밋 메시지 규칙(Feat, Docs, Bug, Fix 등)을 정해둔 것도 실무에서 아주 중요한 습관이에요.
- `.gitignore`에 `node_modules/`, `dist/`, `build/`를 넣은 것도 잘 했어요!

### 문서화

- `Backend-API.md`에 API 명세서를 잘 정리해두었고, `docs/` 폴더에 가이드 문서도 준비해뒀어요. 문서를 잘 정리해두면 팀원들이 프로젝트를 이해하는 데 큰 도움이 돼요.

## 3. 개선하면 좋을 점

### 3-1. 모든 src 파일이 비어 있어요 (중요도: 높음)

**현재 상태:** `src/` 아래 HTML 14개, CSS 14개, JS 13개 — 총 41개 파일이 모두 0줄(빈 파일)이에요.

```
src/landing/landing.html    → 0줄
src/account/login/login.html → 0줄
src/API/accountAPI/login.js  → 0줄
... (모든 파일이 동일)
```

**왜 개선하면 좋을까?**

파일 구조를 미리 잡아두는 건 좋은 접근이지만, 빈 HTML 파일은 **Vite 빌드 시 에러**가 발생할 수 있어요. 또한 `vite.config.js`에서 이 파일들을 entry point로 등록해뒀기 때문에, 빌드하면 빈 페이지들이 생성돼요.

**추천:** 각 HTML 파일에 최소한의 기본 구조(boilerplate)를 먼저 넣어주세요.

```html
<!-- 예시: src/landing/landing.html -->
<!doctype html>
<html lang="ko-KR">
  <head>
    <meta charset="UTF-8" />
    <meta name="viewport" content="width=device-width, initial-scale=1.0" />
    <title>Bui1d-BOX - 랜딩</title>
    <link rel="stylesheet" href="./landing.css" />
  </head>
  <body>
    <h1>랜딩 페이지</h1>
    <script type="module" src="./landing.js"></script>
  </body>
</html>
```

### 3-2. index.html이 참조하는 파일들이 존재하지 않아요 (중요도: 높음)

**현재 상태:**

```html
<!-- index.html 8~15번째 줄 -->
<link rel="shortcut icon" href="/favicon/likelion.ico" type="image/x-icon" />
<link rel="icon" href="/favicon/likelion.svg" type="image/svg+xml" />
<link rel="preload" as="font" href="/font/suit/SUIT-Variable.woff2" />
<link rel="stylesheet" href="/font/suit/SUIT-Variable.css" />
<link rel="stylesheet" href="/src/common/index.css" />
```

**왜 개선하면 좋을까?**

이 파일들이 실제로 프로젝트에 없어요:
- `/favicon/likelion.ico`, `/favicon/likelion.svg` — favicon 폴더 자체가 없어요
- `/font/suit/SUIT-Variable.woff2`, `/font/suit/SUIT-Variable.css` — font 폴더가 없어요
- `/src/common/index.css` — common 폴더가 없어요

브라우저에서 열면 개발자 도구 콘솔에 **404 에러**가 여러 개 뜰 거예요. 존재하지 않는 파일을 참조하면 페이지 로딩이 느려지고, 의도한 스타일이 적용되지 않아요.

**추천:** Vite 프로젝트에서 정적 파일(favicon, font 등)은 `public/` 폴더에 넣어야 해요.

```
Bui1d-UP/
├── public/                    ← 이 폴더를 만들고
│   ├── favicon/
│   │   ├── likelion.ico
│   │   └── likelion.svg
│   └── font/
│       └── suit/
│           ├── SUIT-Variable.woff2
│           └── SUIT-Variable.css
```

그리고 `src/common/index.css` 파일도 만들어주세요:

```
src/
├── common/
│   └── index.css              ← 공통 스타일 파일
```

### 3-3. ESLint 설정에 React 플러그인이 들어있어요 (중요도: 중간)

**현재 상태:**

```js
// eslint.config.js 4~5번째 줄
import react from "eslint-plugin-react";
import reactHooks from "eslint-plugin-react-hooks";
```

```js
// eslint.config.js 20~22번째 줄
plugins: {
  react,
  "react-hooks": reactHooks,
},
```

**왜 개선하면 좋을까?**

이 프로젝트는 **바닐라 JavaScript** 프로젝트인데, ESLint 설정에 **React 플러그인**이 들어있어요. React는 이 프로젝트에서 사용하지 않는 라이브러리예요.

- `npm install`을 하면 React 관련 ESLint 패키지를 설치하려다가 에러가 날 수 있어요
- ESLint를 실행하면 불필요한 React 관련 규칙이 적용돼요

이건 아마 다른 프로젝트의 설정을 복사해온 것 같은데, 바닐라 JS 프로젝트에 맞게 정리하면 좋겠어요.

**추천:**

```js
// eslint.config.js — React 부분 제거한 깔끔한 버전
import js from "@eslint/js";
import globals from "globals";

export default [
  js.configs.recommended,

  {
    files: ["**/*.js"],
    languageOptions: {
      ecmaVersion: "latest",
      sourceType: "module",
      globals: {
        ...globals.browser,
      },
    },
    rules: {
      /* 기본 코드 품질 */
      "no-unused-vars": "warn",
      "no-console": "warn",
      "no-debugger": "error",

      /* JS 스타일 */
      eqeqeq: "error",
      curly: "error",
      "prefer-const": "error",

      /* 기타 */
      "no-var": "error",
    },
  },
];
```

### 3-4. index.html의 인라인 스타일 (중요도: 낮음)

**현재 상태:**

```html
<!-- index.html 17번째 줄 -->
<body style="margin: 50px">
```

**왜 개선하면 좋을까?**

인라인 스타일(`style="..."`)은 나중에 수정하기 어렵고, CSS 파일에서 스타일을 한눈에 관리할 수 없게 돼요. 특히 공통 CSS 파일(`/src/common/index.css`)을 이미 연결하려고 했으니까, 거기에 스타일을 넣는 게 더 좋아요.

**추천:**

```html
<!-- index.html — 인라인 스타일 제거 -->
<body>
```

```css
/* src/common/index.css — 공통 스타일로 이동 */
body {
  margin: 50px;
}
```

### 3-5. .gitignore에 추가하면 좋은 항목들 (중요도: 중간)

**현재 상태:**

```

**왜 개선하면 좋을까?**

현재 3개만 있는데, 실무에서는 더 많은 항목을 추가하는 게 좋아요. 특히 `.env` 파일은 API 키 같은 민감한 정보가 담길 수 있어서 **반드시** 제외해야 해요.

# 의존성

# 빌드 결과물

# 환경 변수 (API 키 등 민감 정보)

# 에디터 설정
.vscode/
.idea/
*.swp
*.swo

# OS 파일
.DS_Store
Thumbs.db
```

### 3-6. PR 템플릿의 체크박스 문법 오류 (중요도: 낮음)

**현재 상태:**

```markdown
<!-- .github/PULL_REQUEST_TEMPLATE.md -->
- [] 새로운 기능 추가
- [] 버그 수정
```

**왜 개선하면 좋을까?**

GitHub 마크다운에서 체크박스를 만들려면 `[ ]`(대괄호 안에 **공백**)을 써야 해요. `[]`처럼 공백 없이 쓰면 체크박스가 나타나지 않고 일반 텍스트로 보여요.

**추천:**

```markdown
- [ ] 새로운 기능 추가
- [ ] 버그 수정
```

### 3-7. package.json에 프로젝트 이름과 설명이 없어요 (중요도: 낮음)

**현재 상태:**

```json
{
  "type": "module",
  "devDependencies": {
    "eslint": "^10.1.0",
    "jest": "^30.3.0",
    "vite": "^8.0.3"
  },
  "scripts": { ... }

```

**왜 개선하면 좋을까?**

`name`, `version`, `description` 필드가 없어요. 이 정보가 있으면 프로젝트가 뭔지 한눈에 알 수 있고, 나중에 배포하거나 팀원이 볼 때도 도움이 돼요.

**추천:**

```json
{
  "name": "bui1d-up",
  "version": "0.1.0",
  "description": "바닐라 JS + Vite 프론트엔드 프로젝트",
  "type": "module",
  ...
}
```

### 3-8. 각 페이지에 JavaScript 파일이 없는 페이지가 있어요 (중요도: 중간)

**현재 상태:** `src/landing/` 폴더만 `landing.js`가 있고, 나머지 페이지 폴더에는 JS 파일이 없어요.

```
src/account/login/       → login.html, login.css (JS 없음)
src/account/signup/      → signup.html, signup.css (JS 없음)
src/main/main_list/      → main_list.html, main_list.css (JS 없음)
src/main/detail/         → detail.html, detail.css (JS 없음)
src/main/genre_more/     → genre_more.html, genre_more.css (JS 없음)
src/mypage/              → mypage.html, mypage.css (JS 없음)
src/paragraph/upload/    → upload.html, upload.css (JS 없음)
src/paragraph/edit/      → edit.html, edit.css (JS 없음)
```

**왜 개선하면 좋을까?**

`src/API/` 폴더에 API 호출 코드를 분리해둔 건 좋은데, 각 페이지에서 이 API를 호출하려면 페이지별 JS 파일이 필요해요. 예를 들어 로그인 페이지에서 `login.js`가 있어야 폼 제출 시 API를 호출할 수 있어요.

**추천:** 각 페이지 폴더에 해당 페이지의 JS 파일을 만들어주세요.

```
src/account/login/
├── login.html
├── login.css
└── login.js     ← 추가! (여기서 @/API/accountAPI/login.js를 import)
```
